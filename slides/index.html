Architecture

The project is easy to defend because the control points are explicit.

The UI, memory, local guardrails, search, and response handling are separated cleanly. That separation makes the system easier to explain, test, and extend.

~25s
Runtime flow
1
Web UI / CLI
Two entry points, one shared tutoring core.
2
ConversationManager
Tracks multi-turn history, truncation, and academic level.
3
Local guardrails
Quickly refuse unsafe or out-of-scope input before expensive model work.
4
Optional search
Search evidence is reviewed and sanitized before use.
5
ResponseHandler
Normal mode or Strict mode produces the final reply.
Why this matters
  • Clear scope: all subject logic is centralized in settings and prompts.
  • Reliable refusal: obvious bad requests can be blocked locally.
  • Testable safety: guardrails, strict behavior, and search filtering all have unit tests.
  • Good demo value: each stage maps to a story you can explain quickly during presentation.

Guardrails ①

The local prefilter is the most important engineering choice in the project.

It does more than keyword blocking. It normalizes obfuscated input, distinguishes academic intent from daily-life services, and protects the subject boundary before the model is called.

~40s
What the prefilter does
1
Decode first
Base64, ROT13, Morse, invisible characters, and light obfuscation are normalized before classification.
2
Catch jailbreak patterns
It blocks prompt leakage, roleplay scaffolds, and “ignore previous instructions” style attacks.
3
Enforce homework-only scope
Clearly disallowed subjects like biology or programming are refused.
4
Avoid false positives
A real conceptual question about climate should pass; a real travel-booking request should not.

Guardrails ②

Strict mode makes safety fail-closed: review the input, generate a draft, then audit the draft.

This is the second major guardrail layer. The model can help generate content, but it does not control the final decision on its own.

~40s
Strict-mode flow
0
Local prefilter
Obvious bad cases are blocked immediately.
1
Input Review
A reviewer returns structured JSON with decision and summary.
2
Answer Generation
A candidate answer is created only after approval.
3
Final Audit
The draft must pass a final check before the user sees it.
Why it is reliable
Input Review
allow
Malformed reviewer output should not pass silently; tests require refusal.
Generation
retry
If the answer is empty or rejected, the system can retry with corrective feedback.
Search review
filter
Irrelevant or prompt-like external snippets are filtered before they reach the UI or the generator.
Evidence · /tests/test_response_handler.py · /tests/test_search.py
Short line: “Generation is useful here because it is fenced in by review and audit.”
